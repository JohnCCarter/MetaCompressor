# MetaCompressor Delta Encoding Benchmark

Generated by: `python benchmarks/bench_corpus.py`  
Platform: Linux / Python 3.12  
MetaCompressor version: 0.1.0  
Date: 2026-05-01

---

## Goal

Measure whether delta encoding closes the gap between MC corpus mode and TAR+ZSTD.

---

## Method

`bench_corpus.py` was extended to run four compression methods on the same
synthetic corpus (35 files, 1.16 MB raw):

| # | Method | Description |
|---|--------|-------------|
| 1 | ZSTD per file | Each file independently compressed at level 3 |
| 2 | TAR + ZSTD (baseline) | Deterministic tar, then ZSTD level 3 |
| 3 | MC compress-dir (no delta) | Shared chunk dict, no intra-chunk delta |
| 4 | MC compress-dir (+ delta) | Shared chunk dict **with** delta encoding enabled |

The `use_delta` parameter added to `compress_corpus()` lets the benchmark
toggle delta encoding cleanly.  The CLI gains a matching `--no-delta` flag.

## Analysis

### Delta encoding impact: zero on this corpus

Delta encoding found **no near-duplicate chunk pairs** in the synthetic log
corpus.  Each log file is generated with a different random seed, so individual
4 KB chunks do not exceed the 80 % byte-similarity threshold required for a
delta to be accepted.  As a result:

- The compressed sizes for "no delta" and "+ delta" are identical (187.5 KB).
- The "+ delta" run is **significantly slower** (2.33 s vs 0.008 s) because
  `find_similar_chunk` scans up to 64 recent same-size chunks for every new
  chunk, even when none qualify — pure overhead with zero payoff.

### Why MC still trails TAR+ZSTD

TAR+ZSTD wins by 3.3 KB (1.8 %) because it hands zstandard a single
concatenated byte stream.  Zstandard's sliding-window back-references span
across file boundaries with no per-chunk framing cost.  MC's fixed-chunk
boundaries fragment the input and add msgpack container overhead, which
zstandard cannot recover.

### When delta encoding would help

Delta encoding is most beneficial when the corpus contains **near-identical
files** — for example:

- Timestamped config snapshots differing in only a few fields.
- Rolling log segments where adjacent windows share most of their content.
- Versioned binary blobs with small incremental changes.

On such a corpus the 80 % similarity threshold would be triggered frequently,
delta diffs would be much smaller than raw chunk bytes, and the msgpack payload
fed to zstandard would shrink meaningfully.

---

## Verdict

**NO_EDGE** on the synthetic benchmark corpus.

- Delta encoding does **not** close the gap vs TAR+ZSTD on randomly generated
  log files with no cross-chunk near-duplicates.
- Delta has **zero compression benefit** and adds ~290× slowdown on this corpus.
- The primitive is correct and will benefit corpora with genuinely similar
  chunks; the benchmark corpus simply does not exercise that path.

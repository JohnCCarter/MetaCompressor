## Did MC Beat ZSTD?

**No file**: MC did not beat ZSTD on any tested file. ZSTD produced smaller output in every case.

### Likely Reasons

| File | Analysis |
|------|----------|
| **empty.txt** | Both formats add a fixed header overhead. MC adds ~47 bytes (magic + version + msgpack/zstd envelope) vs. ZSTD's 9-byte stream. No data to compress in either case. |
| **small.txt** | At 150 bytes the MC container overhead (magic, version, msgpack envelope, sequence list) exceeds any compression gain. ZSTD at level 3 still compresses lightly. |
| **repeated.txt** | MC deduplication collapses 50 000 bytes to a single 45-byte chunk + a short sequence list (215 bytes total). ZSTD's LZ77/entropy coding is even more aggressive on pure repetition (65 bytes). |
| **app.log / json_logs.log** | MC cross-chunk deduplication helps here (similar 4 096-byte windows recur), but ZSTD's sliding-window back-references work within *and* across chunk boundaries. MC's fixed-chunk boundary prevents it from exploiting partial-block repetitions that ZSTD catches, leaving MC ~2–4 % larger. |

### Path to Winning

To close the gap MC would need: variable-length chunking (content-defined), delta-encoding within similar chunks, or a stronger entropy coder on the sequence list. The deduplication primitive is sound and produces excellent results on highly repetitive data (0.43 % of original for `repeated.txt`).
